Test Plan for Social Network Application

Overview

This test plan is created to test the social network application, with three main objectives:

  • Manual testing of core functionalities such as user registration, login, post creation, comments, and user interactions.
  • API testing using Postman to validate the backend services.
  • Automated testing using Selenium WebDriver and Java to streamline repeatable tests of the user interface.

Scope of Testing

The tests will cover the following key functionalities:

  • Manual Testing: User registration, login, post creation, likes, comments, profile management, and user interactions.
  • API Testing: CRUD operations (create, read, update, delete) for users and posts using Postman.
  • Automation: Core scenarios such as registration, login, and post creation using Selenium WebDriver and Java.
  • Cross-browser Testing: Testing on Google Chrome, Mozilla Firefox, and Safari.
  • Cross-platform Testing: Testing on macOS and Windows.
  • Performance Testing: Load testing to ensure system performance under high traffic.

What will NOT be tested:

  • Third-party integrations
  • Detailed UI/UX design validation
  • Performance on unsupported browsers or devices

Objectives of Testing

  • Ensure stability and proper functionality of the application across all supported browsers and operating systems.
  • Validate that APIs and the user interface work correctly and return expected results.
  • Automation will help to speed up and enhance the efficiency of testing core functionalities.
  • Manual Testing will cover detailed checks of functionalities that are difficult to automate.

Testing Strategy

  • Manual Testing of key functionalities: user registration, login, post creation, comments, likes, and user interactions.
  • API Testing with Postman to validate backend requests and responses.
  • Automation with Selenium WebDriver to test key scenarios such as registration, login, and post creation.
  • Cross-browser Testing to ensure compatibility across all supported browsers.
  • Performance Testing to verify system stability under heavy load.

Entry Criteria

  • All test environments should be fully set up and accessible.
  • All test data for the different test scenarios should be available and validated.
  • All necessary tools (Postman, Selenium, browsers, etc.) should be installed and configured.
  • Test cases should be reviewed and approved by the team.

Exit Criteria

  • All planned test cases have been executed.
  • All critical and high-severity defects have been resolved and verified.
  • No open critical defects exist.
  • Test coverage has reached at least 90% for all core functionalities.
  • Automation scripts should pass without major failures.
  • Performance tests should demonstrate stable behavior under the expected load.

Roles and Responsibilities

The testing team consists of Ivan Velchev, Armine Stoykova, and Gabriel Georgiev. The division of tasks is as follows:

  • Manual Testing: All testers will share manual testing tasks equally.
  • API Testing: Each tester will take a portion of the API testing using Postman.
  • Automation: Test automation tasks will be divided among the three testers.
  • Cross-browser Testing: Each tester will test on different browsers.
  • Test Management: All testers will be responsible for reporting bugs and updating progress in Jira/Trello.

Test Scenarios

Key manual testing scenarios include:

  • User Registration: Validate registration with valid and invalid inputs.
  • User Login: Validate login with valid and invalid credentials.
  • Post Creation: Test post creation with valid and invalid inputs.
  • Anonymous Users: Validate actions that anonymous users can or cannot perform.
  • Authorized Users: Validate functionalities available to logged-in users.
  • Search Function: Validate search capabilities for authorized and unauthorized users.
  • Profile Management: Test user profile updates and settings.
  • Interactions with Other Users: Validate user connections, likes, and comments.
  • Administrator Rights: Test administrative functionalities for posts and users.
  • Optional Features: Validate optional features like ID verification.

REST API Functionality Overview

1. Users

CRUD Operations:

  • POST: Register a new user.
  • GET: Retrieve a user's information.
  • PUT: Update user information.
  • DELETE: Delete a user from the system.

Add/Approve Friend:

  • POST: Send a friend request to another user.
  • PUT: Approve a pending friend request.

Get News Feed:

  • GET: Retrieve the latest posts from the user's connections, ordered chronologically.

2. Posts

CRUD Operations:

  • POST: Create a new post.
  • GET: Retrieve a specific post.
  • PUT: Update an existing post.
  • DELETE: Delete a post.

Comment Posts:

  • POST: Add a comment to a specific post.
  • GET: Retrieve all comments on a post.

Like/Unlike Posts/Comments:

  • POST: Like a post.
  • POST: Unlike a post.
  • POST: Like a comment.
  • POST: Unlike a comment.

Exception Handling and Propagation

To ensure that the API handles errors correctly, it is crucial to verify that exceptions are properly caught, logged, and meaningful error messages are returned to the client.

Validation Errors:

  • Send a POST request to /api/users with missing or invalid fields (e.g., missing email or invalid password format).

Resource Not Found:

  • Send a GET request to /api/users/{id} or /api/posts/{id} with a non-existent ID.

Unauthorized Access:

  • Send a request to a protected endpoint (e.g., /api/users/{id}) without authentication or with invalid credentials.

Forbidden Action:

  • Try to perform an action (e.g., delete a user or post) without sufficient permissions.

Internal Server Errors:

  • Simulate an unexpected error (e.g., database connection failure) and send a request to any endpoint.

Exception Logging:

  • Cause an error in the system (e.g., invalid data or unauthorized access). Verify that the exception is logged in the system's error logs with the correct level of detail (stack trace, request data, and timestamp).
